    What an LLM actually does

    One job. Repeated billions of times.

    Given all the text so far, the model assigns a probability to every possible next token and samples one.

    That's it. No retrieval, no reasoning engine, no planning module, no memory beyond the window. Everything else you see in an "agent" is added around the model.

    Tokens — the unit the model sees

    Models don't read characters or words. They read tokens: chunks of text drawn from a fixed vocabulary of roughly 50–100k entries. Common words survive as one token; rarer words get split into sub-word pieces.

    Why this matters: context window size is measured in tokens, not characters or words. A 200k-token context is roughly 150k English words. Every system prompt, every tool result, every file you @-mention — it all eats the same budget.

    The context window

    The model's working memory. It is finite, it is shared, and it is the single most important constraint to design around.

    0 / 200,000 tokens
    raw: 0 compacted: 0
    ⚡ Compaction triggered — context exceeded the window. The harness summarized the oldest items (at ~20% of original size). Detail is lost; the model can still see what was there, just not verbatim.

    Click things to add them to the context:

    When the window fills up, the harness doesn't panic — it compacts: the oldest content gets summarized down to a fraction of its original size. Modern harnesses manage this for you, but the principle holds: everything the model knows in this turn is what's in the window right now.

    The model is the brain. The harness is the hands.

    This is the single most important distinction in this whole lecture.

    The model is the same model whether you talk to it via ChatGPT, Claude Code, opencode, or a curl call to the API. The harness is everything else: the tool loop, the file access, the memory, the command palette, the user interface.

    Swap the harness, keep the model — and the capability changes dramatically. Swap the model, keep the harness — and the output style changes, but the structural capability stays roughly the same.

    Same model, different harness, different agent.

    Most people use a chat UI. The harness turns the same model into something else.

    chat UI

    One-shot Q&A

    You paste a snippet, it replies with text. No files, no execution, no memory between sessions. Most people's mental model of "AI".

    IDE extension

    Inline assist

    Tab-completion on the current line, or a side panel that sees the open files. Powerful for typing, weak for multi-step work.

    Loop + tools

    Read files, run shell, edit, test, iterate. Long-horizon. This is the mode we're talking about today — and the one that compounds.

    An agent harness is a loop: the model produces output, the harness executes the tool calls in that output, feeds the results back, and asks for the next move. This loop is the entire magic. Everything else is polish.

    What goes into the context window

    The model knows only what is currently in the window. You decide what goes in.

    system System prompt — the model's role, constraints, tone ~1–4k tokens
    skills Loaded skill instructions (we'll come back to this) ~0.5–2k tokens each
    history Past messages, tool calls, tool results grows over time
    tools Tool schemas — what the model is allowed to call ~1–5k tokens
    @files Files you've @-mentioned or that the agent has read whatever they need
    user Your most recent message ~tens–hundreds of tokens

    An agent harness composes all of these for you, on every turn. Skills, files, history, tool schemas — they're not magic. They're literally text being prepended to your prompt.

    Skills — packaged instructions, loaded on demand

    A skill is a small bundle: instructions for the model, plus sometimes scripts. The agent loads it when you invoke the name. Click each card to see what it does.

    Skills are how you teach the agent your preferences once and never explain them again. The mattpocock/skills collection is one curated set — a great default, forkable, and you can write your own.

    Agents — persistent, named configurations

    A skill is a tip. An agent is a personality.

    An agent is a named slot with its own system prompt, its own set of allowed tools, and its own model. opencode ships with several by default:

    build

    Default build agent

    Full tool access. Use for: implementing features, fixing bugs, writing tests. The generalist.

    plan

    Read-only planner

    No edit tools. Use for: sketching approaches, reviewing diffs, sanity-checking before you commit. The "let me think about it" mode.

    You can Tab between agents in the opencode TUI, or configure custom agents in opencode.json for specialist roles — a doc-writer agent, a test-runner agent, a security reviewer. They are just files.

    Subagents — agents that spawn other agents

    When the work decomposes, the parent agent can delegate.

    The parent agent decides what to delegate, in parallel, and merges the results.

    Why bother: context isolation. Each subagent has its own clean window, so a long task doesn't pollute the parent's history with thousands of tokens of intermediate work. The parent only sees the final reports.

    Why opencode for this lecture

    • Open source. You can read every line, you can fork it, you can vendor a pinned version into a research group's internal infra. This matters in academia.
    • Model-agnostic. Today we'll use Anthropic or OpenAI via API keys, but the same tool talks to local Ollama models next week. The harness is the asset, not the model.
    • Terminal-native. PhD students live in terminals. opencode is a TUI; it works over SSH; it doesn't need a GUI.
    • Real LSP integration. The model sees your type errors, your test failures, your linter complaints as first-class context — not as a paste.
    • First-class skill support. Reads SKILL.md files, both user-level (~/.config/opencode/skills/) and project-level (.opencode/skills/).

    Matt Pocock's skills — the four failure modes

    The collection is structured around the four ways agents fail. Each skill is the fix.

    #1
    The agent didn't do what I want

    You think you told it. It thinks you told it something else. The misalignment problem.

    Fix → /grill-me · /grill-with-docs (interview you until the spec is sharp)
    #2
    The agent is way too verbose

    It uses 20 words where 1 will do. No shared vocabulary, no shared domain language.

    Fix → CONTEXT.md (a shared-language file the agent reads on every turn)
    #3
    The code doesn't work

    No feedback loop, so the agent is flying blind. It can't tell working from broken.

    Fix → /tdd (red-green-refactor) · /diagnose (for hard bugs)
    #4
    We built a ball of mud

    Agents accelerate code production — and they accelerate entropy just as fast.

    Fix → /zoom-out · /to-prd · /to-issues (deliberate architecture investment)

    Step 1 — install opencode (5 min)

    You probably have it already. If not, this is all you need.

    1

    Run the install script

    One-liner, downloads a static binary. Works on macOS, Linux, and WSL.

    # macOS / Linux / WSL
curl -fsSL https://opencode.ai/install | bash
    2

    Or: install via your package manager

    npm install -g opencode-ai          # Node.js
brew install anomalyco/tap/opencode # macOS / Linux
pacman -S opencode                  # Arch
    3

    Configure a model provider

    opencode needs an API key for some model. Easiest path: /connect in the TUI and follow the prompts. Or set ANTHROPIC_API_KEY / OPENAI_API_KEY in your shell.

    4

    Verify

    $ opencode --version
1.x.x

    Step 2 — install Matt Pocock's skills (5 min)

    One command. The skills land in ~/.config/opencode/skills/.

    1

    Run the universal installer

    Works for opencode, Claude Code, Cursor, Windsurf, Codex, and 15+ other agents — same syntax, different destination.

    $ npx skills@latest add mattpocock/skills

    You'll be prompted to (1) pick which skills to install, and (2) which agents to install them on. Select opencode. You don't have to take all of them — for this lecture, the ones you actually need are grill-me, tdd, to-prd, to-issues, and setup-matt-pocock-skills.

    2

    What just happened

    opencode now has access to those skills as commands. Type / in the TUI to see them — they'll show up as /grill-me, /tdd, etc.

    Step 3 — the one-time setup (3 min)

    Some of the engineering skills need to know where you keep things. Run this once per repo.

    1

    Run /setup-matt-pocock-skills inside opencode

    It will quiz you — three or four short questions:

    • Which issue tracker do you use? (GitHub Issues / Linear / local markdown / none)
    • Which labels do you apply when triaging? (e.g. bug, enhancement, chore)
    • Where do you want domain docs saved? (e.g. docs/)

    For the lecture, pick local markdown for the tracker (we're not wiring up GitHub), and accept the default labels. The setup writes a small AGENTS.md section so the other skills know where to put things.

    Step 4 — /grill-me on a small change (12 min)

    Before you ask the agent to build, the agent asks you questions. One at a time. Until the spec is sharp.

    1

    Pick a small change you actually want

    Something that has a real spec ambiguity in it. Good candidates on the sample app:

    • Add a deadline field to todos, with overdue-highlighting in the frontend.
    • Add a tag field and a filter UI.
    • Replace the file-based storage with SQLite — without breaking the existing API contract.
    2

    Trigger the grilling

    # inside opencode:
/grill-me I want to add a deadline field to todos with
overdue-highlighting in the frontend.

    The skill will ask you between 5 and 15 questions. Answer honestly. If you don't know, say "I don't know, pick a sensible default and tell me what you chose."

    3

    What you get out

    A precise spec, written down, that you can hand to /to-prd or directly to a build agent. The interview is the deliverable — not the code.

    Step 5 — /tdd on the missing API tests (15 min)

    Sample app has zero tests for the API layer. The TDD skill forces a red-green-refactor loop on the agent.

    1

    Trigger

    # inside opencode:
/tdd We need test coverage for the API layer in
@app/main.py. Use FastAPI's TestClient. Cover:
list, create (valid + invalid), read (existing + 404),
patch, and delete (existing + 404). The test file
does not exist yet — create @tests/test_api.py.
    2

    Watch the loop

    The skill will:

    1. Write a failing test for the smallest behaviour (red).
    2. Run it, confirm it fails for the right reason.
    3. Write the minimum code to pass (green).
    4. Refactor if there's duplication.
    5. Repeat, one vertical slice at a time.

    If the agent tries to skip a step or write all tests at once, the skill pushes back. Let it — that's the feedback the model needs.

    Step 6 — /to-prd from your grilled spec (5 min)

    Take the spec from /grill-me and synthesise a PRD. No interview this time — just a write-up.

    1

    Trigger

    # inside opencode:
/to-prd

    It will look at the current conversation context — including your /grill-me answers — and synthesise a PRD.

    2

    Inspect the output

    The skill asks one question first: which modules are you about to touch? Answer honestly — that's how the skill narrows the PRD scope. Then it writes a docs/ doc containing:

    • the why in 2–3 sentences
    • the user-visible behaviour
    • the acceptance criteria as a checklist
    • any decisions worth remembering (→ ADR)
    • any new vocabulary the project needs (→ CONTEXT.md update)

    Step 7 — /to-issues decomposes the PRD (5 min)

    A PRD is too big to be one task. This skill slices it into small, independently-grabbable issues — each one a vertical slice that ships value on its own.

    1

    Trigger

    # inside opencode:
/to-issues

    It will read the PRD you just generated, quiz you briefly on what counts as a slice, and produce a list of issues. Each one has:

    • a title in plain language
    • an acceptance criteria block
    • a label from your triage vocabulary
    • any dependencies on other issues

    Since we picked "local markdown" for the tracker in step 4, the issues land as docs/issues/NNN-slug.md files. Commit them.

    Step 8 — the Ralph loop (10 min)

    An outer loop picks the next open issue; an inner agent solves it; repeat until the queue is empty. Click Run to watch.

    Issue queue

    In real life, the outer loop is a shell script or a tiny Python driver that opencode-invokes once per issue and checks the result. The pattern's name comes from the ghuntley/ralph reference implementation, which targets Claude Code specifically — but the pattern is universal.

    What to try next

    You've used five skills. There are twenty more in the collection. Here are four worth pulling on next.

    1
    Take a real bug to /diagnose

    Find something broken in a project you actually care about. Run the diagnose skill. Notice the red-green discipline it imposes — the model is forbidden from guessing before reproducing.

    2
    Run /zoom-out on an unfamiliar file

    Open a project you didn't write. Pick a file that scares you. Ask the agent to explain it in the context of the whole system. Compare to reading the file cold — the framing is the value.

    3
    Write a CONTEXT.md for a project you own

    Pick a real codebase. Spend 30 minutes distilling its jargon into a one-page shared language. Watch the next session of opencode against that project become twice as productive.

    4
    Wire opencode against a local model

    Install Ollama, pull a small model, point opencode at it via opencode.json. Notice the capability drop — and notice that for narrow tasks, the drop is smaller than you expected.
