Tool schemas — the menu, written in JSON

Every tool the model can use is described in text. The description is sent on every turn. The model "calls" a tool by emitting JSON that matches the schema — the harness parses it and runs the real thing.

A tool definition in JSON Schema:

{
  "name": "read",
  "description": "Read the contents of a file at the given path.",
  "parameters": {
    "type": "object",
    "properties": {
      "filePath": {
        "type": "string",
        "description": "Absolute path to the file to read."
      },
      "offset": { "type": "number", "description": "Line to start from." },
      "limit":  { "type": "number", "description": "Max lines to return." }
    },
    "required": ["filePath"]
  }
}
cost

You pay every turn

Every tool description is re-sent on every turn. 20 tools × 200 tokens ≈ 4k tokens of overhead before the user says a word. This is why "more tools" is not a feature.

visibility

The model picks from what's in the prompt

If a tool isn't in the schema, the model cannot call it. Disable a tool, and that whole class of action is gone. No prompt injection can re-enable it.

hallucination

Models invent tool names

Without a schema, the model will confidently call run_tests or fix_bug that don't exist. The schema is what grounds it in the real interface.

Good tool descriptions read like a tiny API doc. A vague "do something useful" description is a vague tool. The harness re-pays your writing effort on every single turn.
